Ship Forge integrations without leaving the docs.

Forge DevTools gives teams a typed API client, live tunnel, deploy validator, and event debugger for internal platform tooling. This skeleton is tuned for quick comprehension, copy-ready setup, and dense technical reading.

Quickstart terminal 
# 1. Create a Forge-ready tool
npm create forge-app@latest my-tool -- --template docs-ready

# 2. Add your project token
cd my-tool
forge auth login --workspace acme-platform

# 3. Start the local event tunnel
forge dev --endpoint https://api.forge.dev/v1

Concepts

The model is small: tools, runs, events, checks.

Forge treats every internal tool as a versioned runtime contract. A run executes that contract, emits events, and produces deploy checks that can block releases before broken integrations reach production.

  1. Register a tool contract

    Declare inputs, permissions, outbound endpoints, and owner metadata in `forge.config.ts` so the dashboard can diff changes.

  2. Stream runtime events

    Send build, preview, deploy, and user action events to the ingestion API with a `forge-run-id` correlation header.

  3. Gate risky deploys

    Use checks for missing secrets, schema drift, stale tunnel URLs, permission expansion, and webhook signature failures.

i
Production default

Forge rejects unsigned webhooks and client tokens older than 90 days unless the workspace explicitly enables compatibility mode.

REST API

Endpoint examples built for copy and paste.

The examples below use the stable `https://api.forge.dev/v1` endpoint. Sandbox workspaces can switch to `https://sandbox.forge.dev/v1` without changing request payloads.

  • GET /workspaces/{workspace_id}/tools List registered tools
  • POST /runs Create a correlated run
  • POST /runs/{run_id}/events Append runtime events
  • GET /deploys/{deploy_id}/checks Read deploy blockers
Create a run with the TypeScript SDK
import { ForgeClient } from "@forge-devtools/sdk";

const forge = new ForgeClient({
  token: process.env.FORGE_TOKEN,
  workspace: "acme-platform"
});

const run = await forge.runs.create({
  tool: "deploy-console",
  version: "2026.05.18",
  actor: "buildkite:release-8421",
  metadata: { region: "eu-central-1", commit: "8fb31ad" }
});

await forge.events.append(run.id, {
  type: "deploy.check.completed",
  status: "passed",
  duration_ms: 421
});

Events

Stream events with explicit ownership.

Forge event streams are append-only. Each event needs a type, timestamp, run identifier, and actor. Additional payload fields are schema-checked against your tool contract before they appear in the debugger.

Signed event payload
{
  "type": "preview.url.ready",
  "run_id": "run_01HX2PAG4P7JN2MS6E8M8Y19R2",
  "actor": "github-actions:frontend-release",
  "timestamp": "2026-05-26T13:02:45Z",
  "payload": {
    "url": "https://deploy-console-pr-482.preview.acme.dev",
    "ttl_seconds": 7200,
    "commit": "8fb31ad"
  }
}

Authentication

Use scoped tokens and rotate without downtime.

Tokens are scoped by workspace, environment, and capability. Read-only dashboard tokens cannot append events. Deploy tokens cannot change billing, invite users, or generate long-lived personal keys.

Recommended headers

Send `Authorization`, `forge-run-id`, and `idempotency-key` for mutating requests. Retry safely on `429`, `502`, and `503` only.

Rotation window

Create replacement tokens with `forge tokens rotate --grace 24h` and ship both token IDs until all workers report the new key.

Deploy checks

Block releases before broken tools reach users.

Deploy checks compare the proposed contract to the last healthy production run. The default policy fails on removed fields, permission expansion, unpinned webhooks, and missing owner escalation paths.

GitHub Actions deploy gate
- name: Forge deploy checks
  run: |
    forge checks run \
      --tool deploy-console \
      --environment production \
      --commit $GITHUB_SHA \
      --fail-on permission-expansion,schema-break,missing-owner

Observability

Debug a failed run from the trace, event, or deploy.

The dashboard links every run to OpenTelemetry trace IDs, build logs, deploy checks, and webhook delivery attempts. Support teams can share a redacted run URL without granting repository access.

tr
Trace correlation

Set `FORGE_TRACE_EXPORT=otel` to include `traceparent`, `forge.run_id`, `forge.tool`, and `forge.workspace` attributes on every SDK span.

  • Added `forge checks explain` with direct links to blocking schema fields.

  • Fixed Ruby SDK webhook verification for multi-value signature headers.

  • Improved dashboard redaction for run metadata containing email addresses.
  • Deprecated legacy `x-forge-token` header in favor of standard bearer tokens.
